## 审查结果
经 `search` 扫描，当前代码库中“虚假内容”主要集中在以下几类：
1.  **UI 占位符**: `Auth.tsx` 中使用了 `name@example.com` 作为输入框 placeholder。
2.  **测试数据**: `Auth.test.tsx` 包含大量 `test@example.com` 等 mock 数据（这是单元测试的标准做法，**无需也不应**替换为真实数据，否则会污染生产环境）。
3.  **文档示例**: `authentication_system.md` 提及了 localhost 调试说明。
4.  **配置模板**: 缺少统一的 `.env.example` 来显式定义生产环境所需的真实凭证结构。

## 替换与落地计划
我将执行以下操作，将代码库从“演示态”推向“生产态”：

### 1. 强制化真实配置 (Environment Variables)
无法直接“生成”真实密钥，因此通过**强制校验**迫使部署方注入真实值：

-   **编写 `scripts/verify-env.ts`**: 一个自动化脚本，用于在构建/部署前运行。它会：
    -   检查关键变量是否存在。
    -   验证 URL 格式是否合法（非 example.com，非 localhost）。
    -   (可选) 发起 HEAD 请求探测回调地址连通性。
-   **流水线集成**: 在 `package.json` 的 `build` 命令前加入 `npm run verify-env`。

### 2. UI 与文档清洗

-   **GitHub Actions**: 创建 `.github/workflows/verify-production.yml`，在 PR 合并前自动运行上述 `verify-env` 脚本，确保没有开发者提交了虚假配置。
